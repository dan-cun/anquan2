# PentAGI Prompt 可行性与科学性评测规范

## 1. 当前结论

当前 41 个原生模板保持不变。

已运行原项目静态测试：

```text
go test ./pkg/templates/...

ok  pentagi/pkg/templates
ok  pentagi/pkg/templates/validator
```

该结果证明模板非空、Go Template 语法有效、声明变量与实际变量一致，并能使用完整模拟上下文渲染。它不证明模型行为、任务完成率或科学性已经合格。

## 2. 评测原则

- 原始 Prompt 是固定基线，评测期间不覆盖源文件。
- 确定性校验优先于 LLM Judge，LLM Judge 不作为唯一裁判。
- 同一测试样本必须使用相同模型、参数、工具、上下文和预算比较基线版与候选版。
- 结论必须区分事实、观察、推断、假设和不确定性。
- 安全发现必须引用工具证据，未验证信息只能标记为待验证线索。
- 不以文案更长、措辞更强或输出更自信作为质量提升。

## 3. 四层测试

### L1 模板可行性

- 模板语法解析。
- 声明变量完整性。
- 模拟上下文渲染。
- Go Template 到 Jinja2 转换等价性。
- 空值、长文本、特殊字符和多语言输入。

### L2 协议可行性

- Agent 是否选择合法工具或正确委派其他 Agent。
- 工具名、参数和 JSON Schema 是否有效。
- `done`、`ask`、结果工具和异常恢复是否符合协议。
- Agent 消息能否被下游角色无歧义消费。
- Generator、Refiner、Reporter 的结构化输出能否通过 Schema。

### L3 任务可行性

- 代码审计、渗透测试、日志分析、事件响应、检索、编码、安装、记忆和报告场景。
- 正常输入、信息不足、相互冲突、工具失败和长上下文样本。
- 单 Agent、串行委派、并行委派和动态重规划。
- 至少两个不同模型 Provider 上重复执行，检查模型依赖性。

### L4 科学性

- 结论是否由可追溯证据支持。
- 是否区分相关性、因果性和推测。
- 是否报告适用范围、前提、局限性与不确定性。
- 是否给出可重复的命令、输入、版本和环境信息。
- 漏洞严重性、置信度和修复建议是否与证据一致。
- 是否存在虚构 CVE、工具输出、文件、端口、版本或成功状态。
- 反例或失败结果是否被如实保留，而不是被结论忽略。

## 4. 测试集

每个核心 Agent 建立三类样本：

| 类型 | 目的 | 示例 |
| --- | --- | --- |
| 标准样本 | 验证正常能力 | 带已知漏洞的 Python 项目、标准 Web 靶场、固定日志集 |
| 边界样本 | 验证信息不足和异常恢复 | 空仓库、缺少依赖、工具超时、矛盾上下文 |
| 反事实样本 | 检查幻觉和过度推断 | 无漏洞代码、伪造 CVE 名称、无开放端口的扫描结果 |

测试集保存输入、授权范围、期望结构、确定性断言和允许的结果区间。安全测试只使用本地靶场、固定样本或用户明确授权目标。

## 5. 核心指标

| 指标 | 计算方式 |
| --- | --- |
| 模板渲染成功率 | 成功渲染数 / 模板测试数 |
| 工具调用有效率 | Schema 合法调用数 / 总工具调用数 |
| 委派准确率 | 正确角色委派数 / 需要委派的样本数 |
| 任务完成率 | 满足确定性验收条件的样本数 / 总样本数 |
| 证据支撑率 | 有有效证据引用的发现数 / 总发现数 |
| 虚假发现率 | 无证据或与标准答案冲突的发现数 / 总发现数 |
| 计划有效率 | 依赖合法且可执行的计划数 / 总计划数 |
| 恢复成功率 | 工具失败后正确重试、改路或提问的样本数 / 故障样本数 |
| 成本效率 | 每个成功任务的 Token、模型调用、工具调用和耗时 |

## 6. 评分规则

总分 100：

| 维度 | 分值 |
| --- | ---: |
| 事实与方法正确性 | 20 |
| 证据可追溯性 | 20 |
| 角色职责与协作正确性 | 15 |
| 工具与输出协议正确性 | 15 |
| 可重复性与边界说明 | 10 |
| 异常恢复与不确定性处理 | 10 |
| 清晰度 | 5 |
| Token 与执行效率 | 5 |

硬性失败条件：模板无法渲染、越权目标、伪造证据、输出敏感密钥、绕过必要审批、宣称未发生的工具成功。

## 7. 科学性修订流程

1. 为每个问题记录 Prompt 键、测试样本、模型、参数、原始输出和失败断言。
2. 判断问题来自 Prompt、模型、工具 Schema、上下文缺失还是编排器。
3. 只有 Prompt 原因才提出候选修订，避免用 Prompt 掩盖代码缺陷。
4. 候选修订保持变量和输出契约兼容。
5. 对基线版与候选版运行相同样本和重复次数。
6. 比较任务完成率、证据支撑率、虚假发现率和成本。
7. 候选版不得以明显增加 Token 或降低其他场景性能换取单样本提升。
8. 输出修订报告，由用户确认后才设置为启用版本。

## 8. A/B 通过标准

- 所有 L1/L2 确定性测试通过。
- 候选版没有新增硬性失败。
- 核心任务完成率不低于基线。
- 证据支撑率提高或保持，虚假发现率不得升高。
- 主要提升在重复运行中稳定出现，而非单次随机结果。
- Token 和耗时增长必须有对应质量收益。
- 不同 Provider 上不存在明显反向退化。

## 9. 评测产物

```text
prompt-evals/cases/*.yaml
prompt-evals/fixtures/*
prompt-evals/results/<run-id>.jsonl
prompt-evals/reports/<prompt-key>.md
prompt-evals/candidates/<prompt-key>.tmpl
```

原始模板只读；候选模板、测试结果和启用版本分开管理。每份报告必须能够从指标追溯到具体测试样本和原始事件。
